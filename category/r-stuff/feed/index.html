Messy Column Names

If you are a fairly experienced data scientist using R as your main analytic tool, you must have encountered problems like those:

  • I imported a data file from some other data source and the naming conventions on those variables a messed up. e.g. Spaces in column names that makes referencing then in R harder ( have to use the tick sign “”).

In my example below you can see that I have mixed type of column names

df <- readxl::read_xlsx("test.xlsx")
names(df)

## [1] "full name"          "AGE"                "Salary Expectation"
## [4] "date"               "anotherDate"        "yet another date"

In order to reference the first column full name, I have to wrap the name with “” in R:

df$`full name`

## [1] "michael jordan" "steve jobs"     "el paco"

This is creating some nuances because we shouldn’t really have spaces in variable names.

I know it is fine that we can always do something like

names(df) <- c("full_name", "age", ....)

But what if we have lots of variables? Doing this manually seems impossible.

janitor package

janitor has simple functions for examining and cleaning dirty data. It was built with beginning and intermediate R users in mind and is optimized for user-friendliness. Advanced R users can already do everything covered here, but with janitor they can do it faster and save their thinking for the fun stuff.

We can use clean_names function from janitor package. e.g.

df %>% janitor::clean_names(case = "snake")

## # A tibble: 3 x 6
##   full_name   age salary_expectat~ date                another_date
##   <chr>     <dbl>            <dbl> <dttm>              <chr>       
## 1 michael ~    55            12000 2016-01-31 12:13:14 2017-01-01  
## 2 steve jo~    45            32002 2016-02-01 12:13:14 2017-12-22  
## 3 el paco      35            31313 2016-02-02 12:13:14 2017-01-03  
## # ... with 1 more variable: yet_another_date <chr>

you can change up the output format as well:

clean_df <- df %>% janitor::clean_names(case = "all_caps")

clean_df

## # A tibble: 3 x 6
##   FULL_NAME   AGE SALARY_EXPECTAT~ DATE                ANOTHER_DATE
##   <chr>     <dbl>            <dbl> <dttm>              <chr>       
## 1 michael ~    55            12000 2016-01-31 12:13:14 2017-01-01  
## 2 steve jo~    45            32002 2016-02-01 12:13:14 2017-12-22  
## 3 el paco      35            31313 2016-02-02 12:13:14 2017-01-03  
## # ... with 1 more variable: YET_ANOTHER_DATE <chr>

This is just one simple function from janitor package, but this will save us lots of time when tidying up the data.

Date and time conversion

Date and time objects are always a tricky thing in computer science. There are tons of different way to record time — YYYY-MM-DD-HH-MM-SS, MM-DD-YY, DD/MM/YYYY HHMMSS…and so on. Some of the data you are dealing with are probably already in a good datetime format or POSIXct format in R. But most of the times the aren’t. I have a quick example in the same datafrom df

str(df)

## Classes 'tbl_df', 'tbl' and 'data.frame':    3 obs. of  6 variables:
##  $ full name         : chr  "michael jordan" "steve jobs" "el paco"
##  $ AGE               : num  55 45 35
##  $ Salary Expectation: num  12000 32002 31313
##  $ date              : POSIXct, format: "2016-01-31 12:13:14" "2016-02-01 12:13:14" ...
##  $ anotherDate       : chr  "2017-01-01" "2017-12-22" "2017-01-03"
##  $ yet another date  : chr  "2016-Sep-01 10:11:12" "2016-Sep-02 10:11:12" "2016-Dec-01 10:11:12"

As you can see the last 3 columns:

  • date : a nice POSIXct
  • anotherDate: a character with YYYY-MM-DD format
  • yet another date: a character with month as acronyms, ISO style

How do we unify and clean those fields?

anytime Package

anytime aims to be that general purpose converter returning a proper POSIXct (or Date)
object no matter the input, relying on Boost date_time for the (efficient, performant) conversion. link

One specific function I used a lot from this package is anytime()

library(anytime)

anytime(c("2016-01-01 10:11:12", "2016-01-01 10:11:12.345678"))

]]> https://wwwjk366.github.io/two-simple-yet-powerful-packages-r/feed/ 0 A Non-Modeling Approach Scored Top 50 in a Data Science Competition https://wwwjk366.github.io/a-non-modeling-approach-scored-top-50-in-a-data-science-competition/ https://wwwjk366.github.io/a-non-modeling-approach-scored-top-50-in-a-data-science-competition/#respond Fri, 09 Nov 2018 03:22:16 +0000 https://wwwjk366.github.io/?p=258 The Problem

There is an interesting Kaggle-like competition on [https://www.drivendata.org/competitions/55/schneider-cold-start/]. So basically it is asking you to predict the hourly, daily or weekly consumption of some commercial building given hourly consumption before the prediction period. There is a very good problem description here [https://www.drivendata.org/competitions/55/schneider-cold-start/page/111/].

Of course the first thing comes to my mind was XGBoost (Because most of the competitions on Kaggle are solved using XGBoost). However, the first model I submitted scored absolutely horribly.

They also have a benchmark solution using LSTM [http://drivendata.co/blog/benchmark-cold-start-lstm-deep-learning/] which scored 0.53

My Approach

After I took a deeper look into the problem, I figured why not just using the hourly averages of the building in testing dataset to predict its future consumptions? I know it is not a perfect estimator, but it should be very close for example the 11AM energy consumptions for a building on Monday this week should not be that much different comparing to Monday next week (unless Monday is a public holiday, which we can also adjust for that later).

So there are 3 steps I need to do:

  1. Split daily and weekly prediction into hourly
  2. Make hourly predictions by some kind of averaging history
  3. Sum the daily and weekly prediction back to the original format

Expand Predictions to Hourly

Here I defined a function that will take one row of the data and determine if it need to expand. For example, a daily prediction need to be expanded from 1 row to 24 rows with different timestamp.

expand_time <- function(data, idx) {

  xxx <- data[idx, ]

  if(xxx$prediction_window == "daily"){
    res <- xxx %>%
      # select(-timestamp) %>%
      left_join(data_frame(
        pred_id = xxx$pred_id[1],
        timestamp2 = seq(
          from = xxx$timestamp[1],
          to = xxx$timestamp[1] + hours(23),
          by = "hour"
        )
      ),
      by = "pred_id")
  }else if(xxx$prediction_window == "weekly"){
    res <- xxx %>%
      # select(-timestamp) %>%
      left_join(data_frame(
        pred_id = xxx$pred_id[1],
        timestamp2 = seq(
          from = xxx$timestamp[1],
          to = xxx$timestamp[1] + weeks(1) - hours(1),
          by = "hour"
        )
      ),
      by = "pred_id")

  }else if(xxx$prediction_window == "hourly"){
    res <- xxx %>% 
      mutate(timestamp2 = timestamp)

    }

  return(res)

}

Generate Average Predictions

So my logic to do the averages are very easy:

  1. Try to fill predictions using weekday hourly averages
  2. If cannot find values in step 1, fill in with buildings on/off day hourly average
  3. If cannot find values in step 2, fill in with buildings hourly average
  4. If still cannot find values, fill in with hourly averages of the building under same surface type

In R this will be just some simple left-joins and mutates commands:

vals <- test %>% 
  group_by(series_id, weekday, hour) %>% 
  summarise(mean_consum_wh = mean(consumption, na.rm = TRUE)) 


vals_2 <- test %>% 
  group_by(series_id, hour) %>% 
  summarise(mean_consum_h = mean(consumption, na.rm = TRUE)) 

vals_3 <- test %>% 
  group_by(series_id, offday, hour) %>% 
  summarise(mean_consum_offh = mean(consumption, na.rm = TRUE)) 

vals_alloff <- test %>% 
  group_by(surface, offday) %>% 
  summarise(mean_consum_sf = mean(consumption, na.rm = TRUE)) 

final <- subs_hourly %>%
  left_join(vals, by = c("series_id", "weekday", "hour")) %>%
  left_join(vals_3, by = c("series_id", "offday", "hour")) %>%
  left_join(vals_2, by = c("series_id", "hour")) %>%
  left_join(vals_alloff, by = c("surface", "offday")) %>%
  mutate(consumption = ifelse(is.na(mean_consum_wh),mean_consum_offh,mean_consum_wh),
         consumption = ifelse(is.na(consumption),mean_consum_h,consumption),
         consumption = ifelse(is.na(consumption),mean_consum_sf,consumption)
         )

Generate Submission

Because the submission template actually provided pred_id so all we need to do is taking sum of the value to get back to the submission format:

final_sum <- final %>% 
  group_by(pred_id) %>% 
  summarise(consumption = sum(consumption))

submission_format %>% 
  select(-consumption) %>% 
  left_join(final_sum, by = "pred_id") %>%

Since they do not have a proper documentation, everything is possible here. To be able to solve this we can write a simple retry function that will retry a certain function if the function failed:

retry <- function(expr, isError=function(x) "try-error" %in% class(x), maxErrors=3, sleep=0) {
  attempts = 0
  retval = try(eval(expr))
  while (isError(retval)) {
    attempts = attempts + 1
    if (attempts >= maxErrors) {
      msg = sprintf("retry: too many retries [[%s]]", capture.output(str(retval)))
      flog.fatal(msg)
      stop(msg)
    } else {
      msg = sprintf("retry: error in attempt %i/%i [[%s]]", attempts, maxErrors,
                    capture.output(str(retval)))
      flog.error(msg)
      warning(msg)
    }
    if (sleep > 0) Sys.sleep(sleep)
    retval = try(eval(expr))
  }
  return(retval)
}

m <-  mongolite::mongo(db = "NBA",
                       collection = "Games",
                       url = "mongodb://localhost:27017")


dates <- seq(as.Date("2016/1/1"), as.Date("2018/8/31"), "days")

for (i in 1:length(dates)) {
  flog.info(paste0("Trying: ", i))
  xxx <- retry(GetGames(Date = dates[i]), maxErrors = 3, sleep = 10)
  if(!is.null(xxx)){
    m$insert(xxx)
    flog.info(paste0("Inserted: ", i))
  }
}

Current arallel Programming Paradigms in R

parallel package

parallel package in R provided a simple Fork-Join programming functionality for R. It follows the below procedure:

This Fork-Join way of programming in R has some pros and cos:

  • Data copied to child on write (handled by OS)
  • Very efficient
  • No Windows support
  • Not as efficient as threads (C, C++, FORTRAN)

Some the biggest problem for me is the data copying part. It is okay if you are running some heavy-computing but light-data code. However, say if your dataset is 10G (which is not that big nowadays), doing a 8x parallel requires copying this 10G data into 8 workers so you should at least have >80G RAM in order to run it, not mentioning the copy-pasting cache part. This can be a real deal-breaker for many problems.

foreach package

Another popular computing package On CRAN (made by Revolution Analytics, now MRO) is the foreach package. It is a single interface for a number of “back-end” packages: doMC, doMPI, doParallel, doRedis, doRNG, etc. Comparing to parallel package, it is:

  • A Manager-Workers approach
  • The Idea: Unify the disparate interfaces
  • Works on all platforms (if back-end does)
  • Write the code once, use whichever back-end you prefer
  • Binary operator, non-R-ish syntax

Above two packages are the most popular one so far in terms of doing parallel computing in R. They all working well but has different kinds of limitations. Memory sharing is one and also running those codes on a real cluster can be a bit tricky. pbdMPI claims to provide solutions

A Look into pbdMPI

pbdMPIlinks is claiming doing real SPMD:

Single Program/Multiple Data (SPMD)

  • SPMD is arguably the simplest extension of serial programming
  • Only one program is written, executed in batch on all processors
  • Different processors are autonomous; there is no manager
  • Dominant programming model for large machines for 30 years
  • Works on single machine and also clusters(e.g. YARN)

Introduction to pbdMPI

  • MPI: Standard for managing communications (data and instructions) between different nodes/computers.
  • Implementations: OpenMPI, MPICH2, Cray MPT, …
  • Enables parallelism (via communication) on distributed machines
  • Communicator: manages communications between processors

The general process for directly -or indirectly -utilizing MPI in SPMD programs goes something like this:

  1. Initialize communicator(s).
  2. Have each process read in its portion of the data.
  3. Perform computations.
  4. Communicate results.
  5. Shut down the communicator(s).

Basic Communicator Wrangling

  • Managing a Communicator: Create and destroy communicators
    • init() Initialize communicator
    • finalize() shut down communicator(s)
  • Rank query: Determine the processor’s position in the communicator
    • comm.rank() “who am I?”
    • comm.size() “how many of us are there?”
  • Print: printing with control over which processor prints.
    • comm.print(x, ...)
    • comm.cat(x, ...)

MPI Operations

  • Reduce
  • Gather
  • Broadcast
  • Barrier

Reduce

  • each processor has a number x; add all of them up, find the largest/smallest, …
  • reduce(x, op='sum') -reduce to one
  • allreduce(x, op='sum') -reduce to all

Gather

  • Gather: Say each processor has a number. Create a new object on some processor(s)
    containing all of those numbers.

    • gather(x.gbd) – only one processor gets result
    • allgather(x.gbd) – every processor gets result

Broadcast and Barrier

  • Broadcast: One processor has a number x.gbd that every other processor should also
    have.

    • bcast(x.gbd)
  • Barrier: “computation wall”; no processor can proceed until all processors can proceed.
    • barrier()

Putting them together

Let’s do an exercise of combining what we learned so far:

Generate 1000 random samples from uniform(0,1) distribution on each worker, calculate the total mean of all samples on worker 0, send global mean back to all workers and calculate variance of the samples for each worker.

code <-  "
library(pbdMPI, quiet = TRUE)
init()
myRank <- comm.rank()
x <- runif(1000, 0, 1)
xx <- gather(x, unlist = TRUE)
if (myRank == 0) {
  mean_xx <- mean(xx)
} else {
  mean_xx <- NULL
}
mean_global <- bcast(mean_xx)
vars <- sum((x - mean_global)^2) / 1000
comm.print(vars, all.rank = TRUE)
finalize()"

writeLines(code, "mpicode.R")
cat(system('mpiexec -np 4 Rscript mpicode.R',intern = TRUE), sep = '\n')

Example 2 — Sample Mean and Variance

mpi.stat <- function(x.gbd) {
  ### For mean (x).
  N <- allreduce(length(x.gbd), op = "sum")
  bar.x.gbd <- sum(x.gbd / N)

  bar.x <- allreduce(bar.x.gbd, op = "sum")
  ### For var(x).
  s.x.gbd <- sum(x.gbd^2 / (N - 1))
  s.x <- allreduce(s.x.gbd, op = "sum") - bar.x^2 * (N / (N - 1))

  list(mean = bar.x, s = s.x)
} # End of mpi.stat()

library(randomForest)
library(mlbench)
library(pbdMPI)

init()
data("LetterRecognition")
comm.set.seed(seed =123)
n <- nrow (LetterRecognition)

## get train data

n_train <- floor(0.8*n)
i_train <- sample.int(n , n_train) # Use 4/5 of the data to train
train <- LetterRecognition[i_train, ]
my.test_rows <- get.jid(n-n_train)
test <- LetterRecognition[my.test_rows, ]

## train random forest
comm.set.seed( seed = 1e6*runif(1), diff=TRUE)

my.k <- floor(500/comm.size())
my.rf <- randomForest(lettr ~ ., train , ntree = my.k ,norm.votes = FALSE)


rf.each <- allgather(my.rf)
rf.all <- do.call(combine, rf.each)

## predict test data
pred <- predict(rf.all, test )
correct <- allreduce(sum(pred == test$lettr ))
comm.cat(" Proportion Correct:", correct/(n - n_train) , "\n")

finalize()

Conclusion

pbdR is amazing in a way that can make R leverage naive MPI library. However it does requires a different way of programming that use have to keep in mind all the different operations(reduce, gather, etc). However it is true Single Program/Multiple Data in practice and once the code has been written, it can run on any config as long as MPI is installed. I tried run the exactly same code on my personal YARN cluster and it worked automatically.

However, MPI did receive some critics over the years, check out this amazing blog post from Jonathan Dursi. The TLDR version is that MPI is much faster than the others because it’s bare metal. Which means lots of coding at lower level, lots of boilerplate, etc. That’s why Spark is taking over the HPC industry so fast in the last couple of years.

But hey, it is still cool, isn’t it?
