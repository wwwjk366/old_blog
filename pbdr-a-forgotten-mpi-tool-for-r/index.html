pbdR — A Forgotten MPI Tool for R

Posted on September 29, 2018 by Michael Yan

Introduction

Couple of months ago I was researching on parallel processing and multi-threading in R. I came across a series of tools that created for running MPI (Message Passing Interface) in R. Although those tools seems unmaintained for a while and not many people are using it, but it provided a computer science way of approach to one of the biggest problem that bugging the current R community â€” Parallel computing. Well for people not familiar with HPC(high performance computing) concept, let’s go over some basic concepts:

What is wrong with R?

  • R is slow
  • Bad programs are slower
  • R is high-level language: one line can touch a lot of data
  • R will not fix bad programming

R community has been booming in the last few years, however the performance issue in R hasn’t been solved, or because it’s from the gene of R itself that cannot be solve directly. It must be very frustrated when you running your R on a super powerful VM on AWS with 64 cores and 128G of RAM but you only see one core is running your R code.

So the nature way of thinking is, how do we parallel computing in R?

Current arallel Programming Paradigms in R

parallel package

parallel package in R provided a simple Fork-Join programming functionality for R. It follows the below procedure:

This Fork-Join way of programming in R has some pros and cos:

  • Data copied to child on write (handled by OS)
  • Very efficient
  • No Windows support
  • Not as efficient as threads (C, C++, FORTRAN)

Some the biggest problem for me is the data copying part. It is okay if you are running some heavy-computing but light-data code. However, say if your dataset is 10G (which is not that big nowadays), doing a 8x parallel requires copying this 10G data into 8 workers so you should at least have >80G RAM in order to run it, not mentioning the copy-pasting cache part. This can be a real deal-breaker for many problems.

foreach package

Another popular computing package On CRAN (made by Revolution Analytics, now MRO) is the foreach package. It is a single interface for a number of â€œback-endâ€ packages: doMC, doMPI, doParallel, doRedis, doRNG, etc. Comparing to parallel package, it is:

  • A Manager-Workers approach
  • The Idea: Unify the disparate interfaces
  • Works on all platforms (if back-end does)
  • Write the code once, use whichever back-end you prefer
  • Binary operator, non-R-ish syntax

Above two packages are the most popular one so far in terms of doing parallel computing in R. They all working well but has different kinds of limitations. Memory sharing is one and also running those codes on a real cluster can be a bit tricky. pbdMPI claims to provide solutions

A Look into pbdMPI

pbdMPIlinks is claiming doing real SPMD:

Single Program/Multiple Data (SPMD)

  • SPMD is arguably the simplest extension of serial programming
  • Only one program is written, executed in batch on all processors
  • Different processors are autonomous; there is no manager
  • Dominant programming model for large machines for 30 years
  • Works on single machine and also clusters(e.g. YARN)

Introduction to pbdMPI

  • MPI: Standard for managing communications (data and instructions) between different nodes/computers.
  • Implementations: OpenMPI, MPICH2, Cray MPT, â€¦
  • Enables parallelism (via communication) on distributed machines
  • Communicator: manages communications between processors

The general process for directly -or indirectly -utilizing MPI in SPMD programs goes something like this:

  1. Initialize communicator(s).
  2. Have each process read in its portion of the data.
  3. Perform computations.
  4. Communicate results.
  5. Shut down the communicator(s).

Basic Communicator Wrangling

  • Managing a Communicator: Create and destroy communicators
    • init() Initialize communicator
    • finalize() shut down communicator(s)
  • Rank query: Determine the processor’s position in the communicator
    • comm.rank() â€œwho am I?â€
    • comm.size() â€œhow many of us are there?â€
  • Print: printing with control over which processor prints.
    • comm.print(x, ...)
    • comm.cat(x, ...)

MPI Operations

  • Reduce
  • Gather
  • Broadcast
  • Barrier

Reduce

  • each processor has a number x; add all of them up, find the largest/smallest, â€¦
  • reduce(x, op='sum') -reduce to one
  • allreduce(x, op='sum') -reduce to all

Gather

  • Gather: Say each processor has a number. Create a new object on some processor(s)
    containing all of those numbers.
    • gather(x.gbd) – only one processor gets result
    • allgather(x.gbd) – every processor gets result

Broadcast and Barrier

  • Broadcast: One processor has a number x.gbd that every other processor should also
    have.
    • bcast(x.gbd)
  • Barrier: â€œcomputation wallâ€; no processor can proceed until all processors can proceed.
    • barrier()

code <-  "
library(pbdMPI, quiet = TRUE)
init()
myRank <- comm.rank()
x <- runif(1000, 0, 1)
xx <- gather(x, unlist = TRUE)
if (myRank == 0) {
  mean_xx <- mean(xx)
} else {
  mean_xx <- NULL
}
mean_global <- bcast(mean_xx)
vars <- sum((x - mean_global)^2) / 1000
comm.print(vars, all.rank = TRUE)
finalize()"
 
writeLines(code, "mpicode.R")
cat(system('mpiexec -np 4 Rscript mpicode.R',intern = TRUE), sep = '\n')

mpi.stat <- function(x.gbd) {
  ### For mean (x).
  N <- allreduce(length(x.gbd), op = "sum")
  bar.x.gbd <- sum(x.gbd / N)
 
  bar.x <- allreduce(bar.x.gbd, op = "sum")
  ### For var(x).
  s.x.gbd <- sum(x.gbd^2 / (N - 1))
  s.x <- allreduce(s.x.gbd, op = "sum") - bar.x^2 * (N / (N - 1))
 
  list(mean = bar.x, s = s.x)
} # End of mpi.stat()

init()
data("LetterRecognition")
comm.set.seed(seed =123)
n <- nrow (LetterRecognition)
 
## get train data
 
n_train <- floor(0.8*n)

my.test_rows <- get.jid(n-n_train)

comm.set.seed( seed = 1e6*runif(1), diff=TRUE)
 
my.k <- floor(500/comm.size())
my.rf <- randomForest(lettr ~ ., train , ntree = my.k ,norm.votes = FALSE)
 
 
rf.each <- allgather(my.rf)
rf.all <- do.call(combine, rf.each)

Conclusion

pbdR is amazing in a way that can make R leverage naive MPI library. However it does requires a different way of programming that use have to keep in mind all the different operations(reduce, gather, etc). However it is true Single Program/Multiple Data in practice and once the code has been written, it can run on any config as long as MPI is installed. I tried run the exactly same code on my personal YARN cluster and it worked automatically.

However, MPI did receive some critics over the years, check out this amazing blog post from Jonathan Dursi. The TLDR version is that MPI is much faster than the others because it’s bare metal. Which means lots of coding at lower level, lots of boilerplate, etc. That’s why Spark is taking over the HPC industry so fast in the last couple of years.

But hey, it is still cool, isn’t it?
